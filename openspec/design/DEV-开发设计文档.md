## 2. 概述

### 2.1 背景与目标
天鹂可视化运维平台是一个综合性的运维管理平台，旨在为运维团队提供一站式的运维工具。平台采用双层架构设计：
- **OpenSpec（规划层）**：负责需求文档、技术方案、架构决策记录
- **Superpowers（执行层）**：负责代码实现、CI/CD、自动化部署

本文档为平台整体的开发设计文档，涵盖技术架构、核心模块设计和开发规范。

### 2.2 范围
本文档涵盖平台的核心功能模块：
- 部署管理
- 运维检修升级
- 脚本管理
- 配置管理
- 监控
- 日志分析
- 故障诊断
- 镜像管理
- 数据库备份和恢复
- 日常巡检及报告

### 2.3 术语定义

| 术语 | 定义 |
|------|------|
| DevOps | 开发运维一体化 |
| CI/CD | 持续集成/持续部署 |
| ADR | 架构决策记录 (Architecture Decision Record) |
| MTTD | 平均故障检测时间 (Mean Time To Detect) |
| MTTR | 平均故障恢复时间 (Mean Time To Recover) |

| DEPLOY-001 | 一键部署 | 支持选择应用和环境，一键触发部署 | P0 |

| DEPLOY-005 | 健康检查 | 部署前自动执行健康检查 | P1 |

| DEPLOY-007 | 自定义脚本 | 支持自定义部署前/后置脚本 | P2 |

### 4.2 核心流程

#### 4.2.1 一键部署流程

**流程图：**
```
[开始] --> [选择应用] --> [选择环境] --> [选择版本] --> [创建部署任务]
    │                                                                  │
    │                                                                  ▼
    │                                                          [部署前检查]
    │                                                                  │
    │                                                                  ▼
    │                                                          [执行部署]
    │                                                                  │
    │                                                     ┌────────────┐
    └───[失败]◄──[超时/错误]                               │  验证成功？ │
                                                             └─────┬──────┘
                                                                   │ 否
       ┌────────────────────────────────────────────────────────────┘
       │
       ▼
[记录结果] --> [发送通知] --> [结束]
       ▲
       │
[回滚]─┘
```

**详细说明：**
1. 用户在Web界面选择待部署的应用和环境
2. 系统展示可用的版本列表，用户选择部署版本
3. 系统创建部署任务，生成任务ID
4. 执行部署前检查（服务器可达性、资源、端口）
5. 调用部署引擎执行部署，实时推送日志
6. 部署完成后执行验证
7. 验证成功则记录结果并通知用户；失败则自动回滚

### 4.3 功能详细设计

#### 4.3.1 部署引擎 (Deploy Engine)

**职责描述：**
负责执行具体的部署操作，包括拉取镜像、创建容器、启动服务等。

**类/模块设计：**

| 名称 | 类型 | 职责 |
|------|------|------|
| DeployEngine | 服务 | 核心部署引擎 |
| DeployStrategy | 策略 | 部署策略（蓝绿/金丝雀/回滚） |
| HealthChecker | 检查器 | 健康检查 |
| ContainerManager | 管理器 | 容器管理 |

**关键方法：**

| 方法名 | 入参 | 返回值 | 描述 |
|--------|------|--------|------|
| Deploy | ctx, task | error | 执行部署任务 |
| Rollback | ctx, taskID | error | 回滚部署 |
| HealthCheck | ctx, target | bool | 健康检查 |
| GetDeployStatus | ctx, taskID | Status | 获取部署状态 |
| StreamLogs | ctx, taskID | chan Log | 实时日志流 |

**实现逻辑：**
1. 接收部署任务请求
2. 根据部署策略执行部署（普通/蓝绿/金丝雀）
3. 调用容器运行时（Docker/K8s）执行部署
4. 实时输出部署日志
5. 执行健康检查
6. 验证通过后更新任务状态
7. 失败时自动回滚

## 5. 数据结构设计

### 5.1 核心数据结构

```go
// 部署任务
type DeployTask struct {
    ID          int64     `json:"id"`
    AppID       int64     `json:"app_id"`
    AppName     string    `json:"app_name"`
    Environment  string    `json:"environment"` // dev/staging/prod
    Version     string    `json:"version"`
    Strategy    string    `json:"strategy"`   // normal/blue_green/canary
    Status      string    `json:"status"`      // pending/running/success/failed/rollback
    Progress    int       `json:"progress"`    // 0-100
    DeployLog   string    `json:"deploy_log"`
    Result      string    `json:"result"`
    CreatedAt   time.Time `json:"created_at"`
    UpdatedAt   time.Time `json:"updated_at"`
    FinishedAt  *time.Time `json:"finished_at"`
}

// 应用配置
type AppConfig struct {
    ID          int64     `json:"id"`
    AppName     string    `json:"app_name"`
    AppType     string    `json:"app_type"`     // docker/k8s/shell
    RepoURL     string    `json:"repo_url"`
    Dockerfile  string    `json:"dockerfile"`
    ConfigFiles []string  `json:"config_files"` // 配置文件列表
    HealthCheck HealthCheckConfig `json:"health_check"`
    PreScript   string    `json:"pre_script"`   // 部署前脚本
    PostScript  string    `json:"post_script"`  // 部署后脚本
    EnvVars     map[string]string `json:"env_vars"`
    CreatedAt   time.Time `json:"created_at"`
    UpdatedAt   time.Time `json:"updated_at"`
}
```

## 7. 错误处理

### 7.1 错误码定义

| 错误码 | 错误信息 | 处理方式 |
|--------|----------|----------|
| E001 | 应用不存在 | 提示用户检查应用ID |
| E002 | 环境不可用 | 提示配置环境 |
| E003 | 版本不存在 | 提示选择有效版本 |
| E004 | 服务器不可达 | 检查服务器网络 |
| E005 | 部署超时 | 自动回滚并告警 |
| E006 | 健康检查失败 | 自动回滚并告警 |
| E007 | 容器创建失败 | 检查镜像和配置 |
| E008 | 回滚失败 | 人工介入处理 |

### 7.2 异常处理策略
- 所有异常都记录详细日志
- 关键异常自动触发回滚
- 异常信息实时推送给用户
- 连续失败3次自动禁用该应用部署

## 9. 安全设计

### 9.1 认证授权
- 所有API需通过JWT Token认证
- 基于RBAC的权限控制
- 生产环境部署需二次确认
- 敏感操作记录审计日志

### 9.2 数据安全
- 敏感配置加密存储
- 部署凭证不存储明文
- 传输使用HTTPS

## 10. 性能设计

### 10.1 性能指标

| 指标 | 要求 | 说明 |
|------|------|------|
| 部署响应时间 | <2秒 | 任务创建响应 |
| 并发部署数 | ≥10 | 同一时刻支持 |
| 日志推送延迟 | <1秒 | 日志产生到推送 |
| 部署成功率 | >99.9% | 排除用户取消 |

### 10.2 优化策略
- 使用连接池复用数据库连接
- 部署日志异步写入
- 使用Redis缓存热点数据
- 部署任务状态变更发布到消息队列

## 11. 日志设计

### 11.1 日志级别

| 级别 | 使用场景 |
|------|----------|
| DEBUG | 调试信息、详细执行步骤 |
| INFO | 一般信息、状态变更 |
| WARN | 警告信息、可恢复错误 |
| ERROR | 错误信息、部署失败 |

### 11.2 日志格式
```
2024-01-01 12:00:00.000 [INFO] [deploy] [task_id=123] Pulling image...
2024-01-01 12:00:01.000 [INFO] [deploy] [task_id=123] Creating container...
2024-01-01 12:00:02.000 [ERROR] [deploy] [task_id=123] Container creation failed: port already in use
```

## 12. 部署设计

### 12.1 部署架构
- 前端：Nginx 容器化部署
- 后端：Go 应用容器化部署
- 数据库：MySQL 主从部署
- 缓存：Redis 主从部署
- 日志：Elasticsearch 集群部署

### 12.2 启动顺序
1. MySQL → Redis → Elasticsearch
2. API Gateway → 后端服务
3. 前端 Nginx
4. Agent 部署到目标服务器

## 13. 风险与 Mitigation

| 风险 | 影响 | 解决方案 |
|------|------|----------|
| 部署失败导致服务中断 | 高 | 自动回滚机制 |
| 并发部署导致资源争抢 | 中 | 部署队列和锁机制 |
| 镜像拉取超时 | 中 | 镜像预热和重试 |
| 数据库连接池耗尽 | 高 | 连接池监控和告警 |
| 敏感配置泄露 | 极高 | 加密存储和审计 |
